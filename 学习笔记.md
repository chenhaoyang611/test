
## 8. 分支（Branch）——大白话理解

### 什么是分支？

把仓库想象成一本笔记本：

- **main 分支** = 正式版笔记本，记录的都是确定下来的内容
- **新建分支** = 把笔记本复印一份，在复印件上随便写、随便改，不影响正式版
- **合并分支** = 复印件上的内容确认没问题了，抄回正式版笔记本

### 为什么要用分支？

比如你想给项目加一个新功能，但不确定会不会搞崩现有代码。那就开个分支在新功能上折腾，搞好了再合入 main，搞砸了直接删掉分支，main 毫发无伤。

### 常用命令

```powershell
git branch              # 查看所有本地分支
git branch 新分支名      # 基于当前状态创建一个新分支
git switch 分支名        # 切换到某个分支
git switch -c 新分支名   # 创建并切换到新分支（一步到位）
git merge 分支名         # 把指定分支的内容合并到当前分支
git branch -d 分支名     # 删除分支（合并完后清理用）
```

### 一句话总结

> 分支 = 给代码开个平行世界，搞定了再合回来，搞不定就删掉，main 永远稳如泰山。

### 多人协作场景

假设三个人一起做一个水果清单项目：

```
main  ────●────●────●────●────  （稳定的正式版）
            \    \    \
张三分支 ──●──●──●   \    \     （张三加苹果、西瓜）
                      \    \
李四分支 ──────────────●──●─    （李四加草莓、蓝莓）
                           \
王五分支 ───────────────────●    （王五删香蕉）
```

- 张三、李四、王五各开各的分支，互不干扰
- 谁做完了，确认没问题，就 `git merge` 合入 main
- 如果李四和王五同时改了同一行，合并时会提示冲突，商量一下保留谁的就行

### 一句话总结

> 分支就是每个人拿一份代码副本去改，改好了交回来，改的过程中谁也碍不着谁。

## 9. 分支实操——feature & feature1

### 操作流程

```powershell
git branch feature            # 基于 main 创建 feature 分支
git switch feature            # 切换到 feature 分支
# 在 feature 分支上编辑 fruits.txt，加入菠萝、西瓜
git add fruits.txt
git commit -m "新增菠萝和西瓜"

git switch -c feature1        # 基于 feature 创建并切换到 feature1 分支
# 在 feature1 分支上编辑 fruits.txt，加入车厘子、猕猴桃
git add fruits.txt
git commit -m "新增车厘子和猕猴桃"

git switch main               # 切回 main
git merge feature1            # 把 feature1 合并进 main
```

### 合并前各分支内容

| 分支     | 水果                       |
|----------|---------------------------|
| main     | 香蕉、橘子                  |
| feature  | 香蕉、橘子、菠萝、西瓜       |
| feature1 | 香蕉、橘子、菠萝、西瓜、车厘子、猕猴桃 |

### 合并后

feature1 合入 main，Fast-forward 模式（无冲突，一条直线），main 追上 feature1：

| 分支     | 水果                                         |
|----------|---------------------------------------------|
| main     | 香蕉、橘子、菠萝、西瓜、车厘子、猕猴桃         |
| feature  | 香蕉、橘子、菠萝、西瓜                        |
| feature1 | 香蕉、橘子、菠萝、西瓜、车厘子、猕猴桃         |

### 提交历史

```
a6a9557 新增车厘子和猕猴桃  ← main / feature1
e6342f9 新增菠萝和西瓜      ← feature
a367b62 新增橘子
f14ac97 新增香蕉
59a8d13 Initial commit
```

### 新命令速记

| 命令 | 作用 |
|------|------|
| `git switch -c 分支名` | 创建并切换到新分支（`git checkout -b` 的老版写法） |
| `git merge 分支名` | 把指定分支合并到当前分支 |
| `git log --all --oneline --graph` | 可视化查看所有分支的提交线 |

## 10. 回到过去——两种方式

回到历史某个提交，有两种做法：

### 方式一：创建分支（推荐）

```powershell
git switch -c 新分支名 <commit-hash>
# 或者
git branch 新分支名 <commit-hash>
```

> 相当于在历史节点上"插一面旗"，之后可以继续提交、合并，操作安全可追溯。

### 方式二：分离头指针（临时查看）

```powershell
git switch --detach <commit-hash>
```

> HEAD 直接指向某个提交，不挂在任何分支上。
> 这时做的提交"悬空"，一切回分支就容易丢失。
> 适合临时看一眼旧代码，不适合在上面改东西。

### 大白话对比

| 方式 | 比喻 | 适合场景 |
|------|------|---------|
| 创建分支 | 在历史节点插一面旗，后续随便走 | 要基于旧版本继续开发 |
| 分离头指针 | 时光机直接传送到过去，但没有锚点 | 临时翻看旧代码，看完就走 |

### 一句话

> 要改代码就用分支回去；只看不改就用分离头指针。

## 11. 工作区 vs 本地仓库——区别在哪

### 大白话

Git 把同一个项目分成了两个"地盘"：

| 地盘 | 大白话 | 位置 |
|------|--------|------|
| **工作区（Working Directory）** | 你正在看的、正在改的那些文件 | 就是你的项目文件夹，比如 `C:\Users\chenh\Desktop\test` |
| **本地仓库（Local Repository）** | Git 偷偷存档的地方，记录所有历史版本 | 藏在 `.git` 文件夹里 |

### 用一个例子说清楚

```
你打开 fruits.txt，把"香蕉"改成"榴莲"，保存。
此时：
  工作区 fruits.txt → 榴莲        （你刚改的，还没告诉 Git）
  本地仓库 fruits.txt → 香蕉       （Git 存档的还是旧的）
```

运行 `git add` + `git commit` 之后，本地仓库才更新为"榴莲"。

### 中间还有个暂存区

```
工作区  ──git add──▶  暂存区  ──git commit──▶  本地仓库
（你改文件）        （告诉 Git 哪些要存）      （正式存档）
```

### 一句话

> 工作区是你手上正在改的，本地仓库是 Git 帮你存好的历史版本。
> `git add` 和 `git commit` 就是把工作区的内容正式写入本地仓库。
