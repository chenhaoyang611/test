# Git 系统性学习笔记 (6.3)

## 第一讲：Git 底层原理

### 1.1 Git 是什么

**分布式版本控制系统**——每个人电脑上都有完整仓库，断网也能 commit。

### 1.2 数据模型：快照，不是差异

| 传统方式（SVN） | Git 方式 |
|----------------|---------|
| 存文件差异：基础 + 差异1 + 差异2 | 每次 commit 存整个项目的**完整快照** |
| 回退到旧版本要"倒着算" | 直接切到那张快照，瞬间完成 |
| 没变的文件也占空间 | 没变的文件只存一个引用，不重复存 |

> 每次 `git commit` = 给整个项目拍一张照片。

### 1.3 内容寻址：SHA-1 哈希

Git 存任何内容时：
1. 计算内容的 SHA-1 哈希 → 40 位十六进制指纹
2. 以哈希为文件名存入 `.git/objects/`
3. 相同内容 = 相同哈希，只存一份

> 你看到的 `a367b62` 就是完整哈希的前 7 位缩写。

### 1.4 .git 目录结构（当前仓库实测）

```
.git/
├── HEAD              # 指向当前分支：ref: refs/heads/main
├── config            # 仓库配置（remote、用户信息）
├── index             # 暂存区
├── objects/          # 所有数据（核心！）
│   ├── 03/           # 每个提交、每个文件一个对象
│   ├── 05/
│   └── ...
├── refs/
│   ├── heads/        # 分支指针：main, feature, feature1, cherry-demo
│   └── tags/
└── logs/             # 操作历史
```

### 1.5 三棵树：Git 的工作流

```
工作区 (Working)     你正在编辑的文件           ← git add →
暂存区 (Staging)     准备提交的快照（index）     ← git commit →
本地仓库 (Repo)      永久存档的快照（objects）   ← git push →
远程仓库 (Remote)    GitHub 上的仓库
```

### 1.6 一张图总结

```
编辑文件 ──git add──▶ 暂存区 ──git commit──▶ 本地仓库 ──git push──▶ 远程
                        │                       │
                    可撤销                    可回退
                  git restore              git reset / revert
                  --staged
```

## 第二讲：git init / add / commit 深入

### 2.1 git init 做了什么？

```powershell
git init
```

只做一件事：创建 `.git` 目录及其子目录（objects、refs、hooks 等）。

> 此时没有任何 commit，HEAD 指向 `refs/heads/main`，但 main 分支还不存在（第一个 commit 后才会创建）。

### 2.2 git add 做了什么？

```powershell
git add fruits.txt
```

1. 计算文件内容的 SHA-1 哈希
2. 把文件内容压缩后存入 `.git/objects/`（作为 blob 对象）
3. 更新 `.git/index`（暂存区），记录「这个文件要包含在下一次 commit 中」

> `git add` = 把工作区的文件**复制一份快照**到暂存区。

### 2.3 git commit 做了什么？

```powershell
git commit -m "新增香蕉"
```

1. 把暂存区的内容打包成一个**tree 对象**（目录快照）
2. 创建一个**commit 对象**，包含：
   - tree 的哈希（指向目录快照）
   - 父 commit 的哈希（形成链条）
   - 作者、时间、提交信息
3. 把当前分支指针（`refs/heads/main`）指向新 commit

### 2.4 一次 commit 的内部结构

```
commit f14ac97
├── tree: a1b2c3d...      ← 指向当时的目录结构
│   ├── blob: e4f5g6h...  ← .gitignore 的内容
│   └── blob: i7j8k9l...  ← fruits.txt 的内容
├── parent: 59a8d13       ← 上一个 commit
├── author: chenhaoyang611
└── message: "新增香蕉"
```

> 每个 commit 就是一棵树，指向当时所有文件的快照。

### 2.5 完整链条

```
59a8d13  ←── f14ac97  ←── a367b62  ←── e6342f9  ←── ...
  tree         tree         tree         tree
  (空)      香蕉 橘子    香蕉橘子     香蕉橘子
                                    菠萝西瓜
```

> 每次 commit 都存一份完整的文件树。没变的文件只存引用（同一个 blob 哈希），不重复占用空间。

## 第三讲：分支的本质——只是指针

### 3.1 实测：分支就是一个文件

你的仓库里，每个分支本质上只是一个**存了 40 位哈希值的文本文件**：

```
.git/HEAD           → ref: refs/heads/main     （HEAD 指向 main）
.git/refs/heads/main      → 8d95d8b...         （main 指向这个 commit）
.git/refs/heads/feature   → e6342f9...         （feature 指向 新增菠萝和西瓜）
.git/refs/heads/feature1  → a6a9557...         （feature1 指向 新增车厘子）
.git/refs/heads/cherry-demo → 376f8c1...       （cherry-demo 指向 新增葡萄）
```

### 3.2 一张图看懂

```
HEAD → main → 8d95d8b（大幅精简学习笔记）
             ↑
             这就是一个指针，存了一个哈希值
```

```
main   ──▶ 8d95d8b 大幅精简学习笔记
feature ──▶ e6342f9 新增菠萝和西瓜
                    ↑
              只存了 40 个字符！
```

### 3.3 切换分支到底干了什么？

```powershell
git switch feature
```

Git 只做两件事：
1. 把 `HEAD` 的内容改为 `ref: refs/heads/feature`
2. 把工作区的文件替换成 feature 指向的那个 commit 的快照

> 就这么简单！没复制文件、没创建目录，就是改了个指针。

### 3.4 git merge —— 两种模式

#### 模式一：Fast-forward（快进）

当被合入的分支是当前分支的直系后代时：

```
main:    A → B
feature: A → B → C → D

git switch main; git merge feature

结果：main 指针直接跳到 D，一条直线，不产生新 commit
```

你之前 `git merge feature1` 就是 Fast-forward。

#### 模式二：三方合并（3-way merge）

当两个分支分叉了：

```
main:    A → B → C
              \
feature:       D → E

git switch main; git merge feature

结果：
main:    A → B → C → M（新的合并 commit）
              \      /
feature:       D → E
```

> Fast-forward 不产生新 commit；三方合并会产生一个"合并提交"。

### 3.5 合并冲突的本质

当两个分支改了**同一个文件的同一行**，Git 算不出来用哪个，只能扔给你决定：

```
<<<<<<< HEAD
菠萝           ← 当前分支的版本
=======
凤梨           ← 要合进来的版本
>>>>>>> feature
```

> 冲突不是 bug，是 Git 在说"这两边都改了这行，你来拍板用哪个"。

## 第四讲：git reset 三兄弟

### 4.1 一张表说清楚

| 模式 | 命令 | 改了 HEAD/分支？ | 改了暂存区？ | 改了工作区？ |
|------|------|:---:|:---:|:---:|
| `--soft` | `git reset --soft <hash>` | ✅ 移指针 | ❌ 保留 | ❌ 保留 |
| `--mixed` | `git reset <hash>`（默认） | ✅ 移指针 | ✅ 清空 | ❌ 保留 |
| `--hard` | `git reset --hard <hash>` | ✅ 移指针 | ✅ 清空 | ✅ 覆盖 |

### 4.2 大白话

```
假设你提交了 A → B → C，现在想回到 B：

--soft：  指针回到 B，C 的改动还在暂存区 → 可以重新 commit
          相当于"撤回 commit，但留着改动准备重新提交"

--mixed： 指针回到 B，C 的改动还在工作区，但不在暂存区 → 需要重新 add
          相当于"撤回 commit 和 add，改动还在文件里"

--hard：  指针回到 B，C 的改动彻底没了 → 干干净净回到 B
          相当于"就当 C 从没存在过"
```

### 4.3 一张图

```
A → B → C（当前）

git reset --soft B：
A → B                    暂存区：C 的改动还在
                         工作区：C 的改动还在

git reset B（--mixed）：
A → B                    暂存区：空了
                         工作区：C 的改动还在

git reset --hard B：
A → B                    暂存区：空了
                         工作区：空了（C 的改动没了！）
```

### 4.4 你昨天用的

你昨天用的是 `git reset --hard a367b62`，直接丢弃了"新增葡萄"，工作区也干净了。

```powershell
# 三种模式的典型场景
git reset --soft HEAD~1    # "commit 写错了，撤回重写"
git reset HEAD~1           # "add 加多了，撤回重新挑"
git reset --hard HEAD~1    # "这几次全不要了，从头来"
```

## 第五讲：cherry-pick / 分离头指针 / revert

### 5.1 Cherry-pick —— 挑着吃

```powershell
git cherry-pick <commit-hash>
```

**本质**：把某个 commit 的改动"重放"到当前分支上，生成一个新 commit。

```
cherry-demo:  A → B(榴莲) → C(葡萄)
main:          A → D → E

git switch main; git cherry-pick B（榴莲）

结果：
main:          A → D → E → B'(榴莲，新哈希)
cherry-demo:   A → B(榴莲) → C(葡萄)
```

> cherry-pick 不是"移动"提交，是"复制"。原来的提交还在原分支上。

### 5.2 分离头指针（Detached HEAD）

```powershell
git switch --detach <commit-hash>
```

**HEAD 直接指向一个 commit，不通过分支**：

```
正常：HEAD → main → 8d95d8b
分离：HEAD → 376f8c1（没分支啥事）
```

**危险在哪？** 如果这时做了 commit：

```
376f8c1 → 新commit（悬空了！没有分支指向它）
```

一切回 main，那个新 commit 就没法通过分支找到了（虽然还在 objects 里，但容易丢）。

**正确用法**：基于旧 commit 开分支：

```powershell
git switch -c 新分支 <hash>   # 同时建分支，不分离
```

### 5.3 git revert —— 优雅的撤销

`reset` 是"倒退"，`revert` 是"新增一个反向提交"：

```powershell
git revert <hash>   # 创建一个新 commit，内容是"撤销那个 commit 的改动"
```

```
A → B → C（C 是 bug）

git revert C

结果：
A → B → C → D（D 的内容 = 没有 C 时的状态）
```

| 操作 | 历史 | 适合 |
|------|------|------|
| `reset` | 改写了历史 | 本地还没 push 的 |
| `revert` | 保留历史，新增反向提交 | 已经 push 到远程的 |

> 坑：reset 已经 push 的 commit 再 push 会报错（non-fast-forward），需要 `--force`，会把别人的提交也弄没，慎用！

## 第六讲：git diff —— 看懂每一次改了什么

### 6.1 四种对比

| 命令 | 对比什么 |
|------|---------|
| `git diff` | 工作区 vs 暂存区（你改了但还没 add 的） |
| `git diff --staged` | 暂存区 vs 最新 commit（add 了但还没 commit 的） |
| `git diff HEAD` | 工作区 vs 最新 commit（所有还没提交的改动） |
| `git diff <hash1> <hash2>` | 两个 commit 之间的差异 |

### 6.2 读懂输出

```
--- a/fruits.txt        ← 旧文件
+++ b/fruits.txt        ← 新文件
@@ -6,3 +6,4 @@        ← 旧文件第6行起3行 / 新文件第6行起4行
 猕猴桃
 椰子
 榴莲
+芒果                  ← + 表示新增的行
```

```
- 开头 = 删掉的行
+ 开头 = 新增的行
（没有符号）= 没变，给出上下文
```

### 6.3 实用场景

```powershell
git diff                    # commit 前最后确认：我改了什么？
git diff --staged           # commit 前确认：即将提交的是什么？
git diff main..feature      # feature 比 main 多了什么？
git diff HEAD~1             # 和上一个 commit 比改了什么？
```

### 6.4 git diff --stat

只看改了哪些文件、改了多少行，不看具体内容：

```powershell
git diff --stat
# fruits.txt | 1 +     （1 行新增）
```

## 第七讲：git log 高级用法

### 7.1 基础你已经会了

```powershell
git log --oneline --graph --all   # 可视化全部历史
```

### 7.2 过滤

```powershell
git log -3                        # 只看最近 3 条
git log --author="chenhaoyang"    # 只看某个人的提交
git log --since="2026-06-01"     # 只看某天之后

```

### 7.3 格式化

```powershell
git log --oneline                 # 一行一条（你常用的）
git log --pretty=format:"%h %s"   # 自定义格式
  %h = 短哈希    %s = 提交信息    %an = 作者    %ar = 相对时间
```

### 7.4 查看改动

```powershell
git log -p                        # 显示每次提交的具体改动（diff）
git log --stat                    # 显示每次提交改了哪些文件
```

### 7.5 范围查询

```powershell
git log main..feature             # feature 有但 main 没有的提交
git log HEAD~3..HEAD              # 最近 3 次提交
```

## 第八讲：git stash —— 临时保存工作现场

### 8.1 场景

你正在 feature 分支上改代码，改到一半，老板让你紧急修 main 上的一个 bug。你不能 commit（改了一半），也不能丢（白改了）。

### 8.2 命令

```powershell
git stash               # 把工作区和暂存区的改动暂存起来，工作区恢复干净
git stash list          # 查看 stash 列表
git stash pop           # 取出最新 stash 并删除记录
git stash apply         # 取出最新 stash 但保留记录
git stash drop          # 删除某个 stash
```

### 8.3 一句话

> `git stash` = 把改了一半的东西"藏起来"，干完别的再拿出来继续。相当于"存档读档"。

## 第九讲：git rebase —— 把提交线变直

### 9.1 场景

```
main:     A → B → C
               \
feature:        D → E
```

merge 后：

```
main:     A → B → C → M（合并 commit）
               \    /
feature:        D → E
```

有分叉，不美观。rebase：

```
main:     A → B → C → D' → E'
```

一条直线，没有分叉。

### 9.2 命令

```powershell
git switch feature
git rebase main
```

Base 从 B 变成 C，D 和 E 被"接"到 C 后面。

### 9.3 merge vs rebase

| 操作 | 效果 | 何时用 |
|------|------|--------|
| `merge` | 保留真实历史，可能有分叉 | 公共分支（main）合入功能分支 |
| `rebase` | 改写历史，一条直线 | 在自己分支上整理提交，合入前让历史干净 |

> ⚠️ 不要 rebase 已经 push 到远程的公共分支！

## 第十讲：git tag —— 给重要版本打标签

### 10.1 场景

发布了 v1.0，以后想快速找到这个版本的状态。

### 10.2 命令

```powershell
git tag v1.0                          # 在当前 commit 打轻量标签
git tag -a v1.0 -m "正式发布"          # 附注标签（有说明）
git tag                               # 列出所有标签
git switch v1.0                       # 切换到标签对应的状态（会分离 HEAD）
git push origin v1.0                  # 推送标签到远程
git push origin --tags                # 推送所有标签
```

### 10.3 本质

标签和分支一样，也是个指针（存在 `.git/refs/tags/`），但标签是**固定的**，不会随着 commit 移动。分支会动，标签不动。

## 第十一讲：远程协作 —— fetch vs pull

### 11.1 几个概念

| 概念 | 解释 |
|------|------|
| `origin` | 远程仓库的默认别名（可以改） |
| `origin/main` | 远程 main 分支在你本地的**镜像**（存在 `.git/refs/remotes/origin/`） |
| `upstream` | 本地分支"追踪"的远程分支 |

### 11.2 四大远程命令

| 命令 | 干什么 |
|------|--------|
| `git fetch` | 从远程下载更新，但**不合并**，只是更新 `origin/main` 镜像 |
| `git pull` | = `git fetch` + `git merge`，下载并合并（一步到位） |
| `git push` | 把本地 commit 推送到远程 |
| `git clone` | 把远程仓库完整复制到本地 |

### 11.3 fetch vs pull

```
别人的 push ──▶ GitHub（远程 main 更新了）
                      │
               git fetch     ← 只下载，本地代码不变
               git pull      ← 下载 + 自动合并到本地 main
```

```powershell
# 安全的做法
git fetch                   # 先看看远程有什么更新
git diff main origin/main   # 对比一下差异
git merge origin/main       # 确认没问题再合并

# 快捷的做法
git pull                    # 直接拉下来合并
```

> 推荐先用 `fetch` 看一眼，再决定要不要 `merge`。`pull` 一步到位但出了问题不好回退。

### 11.4 多人协作完整工作流

```
1. git clone <url>              # 把仓库下到本地
2. git switch -c my-feature     # 开自己的分支
3. 改代码 → add → commit        # 本地开发
4. git fetch origin             # 看看 main 有没有别人更新了
5. git rebase origin/main       # 把 main 的更新合进来（让历史干净）
6. git push -u origin my-feature # 推上去
7. 在 GitHub 上创建 PR          # 申请合入 main
```

## 第十二讲：git blame + 总复习

### 12.1 git blame —— 每一行是谁写的

```powershell
git blame fruits.txt
```

输出：
```
f14ac97a (chenhaoyang611 2026-06-02) 香蕉
a367b62e (chenhaoyang611 2026-06-02) 橘子
e6342f9b (chenhaoyang611 2026-06-02) 菠萝
...
```

每一行都标注了：commit 哈希、作者、时间。

### 12.2 实用场景

```powershell
git blame fruits.txt -L 5,10      # 只看第 5 到 10 行
git blame -L "/橘子/",+3 fruits.txt  # 从"橘子"开始看 3 行
```

### 12.3 用处

出 bug 了，`git blame` 一看这行是谁什么时候改的，就能找到对应 commit 和上下文。
